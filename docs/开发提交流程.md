# 开发提交流程

这份流程用于后续每次开发新功能、修复问题、提交代码时统一操作。

## 1. 开发方式

默认不要直接在 `main` 上改。

推荐流程：

1. 从 `main` 拉最新代码
2. 新建功能分支
3. 在功能分支开发和自测
4. 推送远程分支
5. 发起 PR
6. 检查通过后再合并到 `main`

示例命令：

```bash
git checkout main
git pull origin main
git checkout -b codex/feature-name
```

如果是修 bug，也建议按同样方式处理：

```bash
git checkout main
git pull origin main
git checkout -b codex/fix-something
```

## 2. 提交前本地检查

当前仓库建议至少做两类检查。

### 后端检查

在 `service/` 目录执行：

```bash
CGO_ENABLED=0 go test ./...
```

说明：

- 这是当前后端最直接、最稳定的回归检查方式
- 主要覆盖路由层和模型层已有测试
- 如果你改了后端逻辑，提交前至少跑一次

### 后台前端检查

在 `background/` 目录执行：

```bash
npm install
npm run build
```

说明：

- 当前仓库还没有纳入 `package-lock.json`，所以这里先使用 `npm install`
- `npm run build` 能帮你提前发现大部分 TS / Vite 构建问题
- 如果只改后台页面，也建议完整跑一次

### 小程序当前情况

`miniprogram/` 目前还没有配置统一的 `test` / `build` 校验脚本，所以暂时没有纳入自动检查。

这意味着：

- 小程序改动目前主要依赖开发者工具手动验证
- 后续如果要提升稳定性，建议补一个最小校验脚本，再接入 CI

## 3. GitHub 自动检查

仓库已经补了 GitHub Actions 工作流：

- 后端自动执行 `CGO_ENABLED=0 go test ./...`
- 后台前端自动执行 `npm install` 和 `npm run build`

触发时机：

- 推送到 `main`
- 推送到 `codex/**` 分支
- 创建或更新 PR

也就是说，你以后只要把分支推上去，GitHub 就会自动帮你跑一遍基础检查。

## 4. 推荐提交节奏

建议按“一个功能点 / 一个修复点”提交，不要一次塞太多内容。

例如：

- 修一个上传 bug，单独一个 commit
- 新增一个后台页面，单独一个 commit
- 调整数据库结构，单独一个 commit

这样做的好处：

- 更容易 review
- 更容易回滚
- 出问题时更容易定位

## 5. 推荐发布节奏

比较稳的方式是：

1. 在功能分支开发
2. 本地跑检查
3. 推送远程分支
4. 在 GitHub 看 CI 是否通过
5. 通过后合并到 `main`
6. 再部署到服务器

注意：

- 代码只在你本地提交，不会自动影响线上
- 必须把新的后端代码部署到服务器，线上才会真正生效
- 前端也是一样，只有重新构建并发布后，线上页面才会更新

## 6. 这次上传问题对应的结论

这次“同名图片上传失败”的修复，虽然已经提交到分支，但如果服务器上还跑的是旧版本后端，线上就还是会报原来的错。

所以这类后端问题的完整生效链路是：

1. 本地修复代码
2. 提交分支
3. 检查通过
4. 合并到主分支
5. 重新部署后端服务

只有走完整条链路，线上才会真正恢复。
